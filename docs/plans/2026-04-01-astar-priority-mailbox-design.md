# A* Priority Registry + Mailbox Relocation Design

**Date:** 2026-04-01
**Status:** Approved
**Scope:** Exodus `.claude/` governance infrastructure

## Context

The A* filesystem search protocol (defined in origin's CLAUDE.md) uses hardcoded priority tables to determine read order during cold starts, loop iterations, and identity realignments. These priorities are static — they never adapt based on actual usage. Additionally, the mailbox files (`prompt-inbox.md`, `prompt-outbox.md`) live inside `.claude/` but logically belong at the project's public interface level.

This design introduces three changes:
1. **Priority frontmatter** on every `.md` file in `.claude/`
2. **A `prioritize` skill** that rescores files each loop iteration and writes a compiled registry
3. **Mailbox relocation** — rename and move inbox/outbox to project root

## 1. Priority Scoring Architecture

### File-Level Frontmatter

Every `.md` file in `.claude/` gets priority metadata in its YAML frontmatter:

```yaml
---
priority: 90       # 1-100 base score, higher = read first
last_read: <ISO>   # updated when file is read by self-scan/loop
read_count: 0      # lifetime reads (informational, not used in scoring)
---
```

### Compiled Registry: `.claude/priorities.md`

Auto-generated by the prioritize skill. Read by loop/self-scan instead of hardcoded tables.

```markdown
# .claude/ Priority Registry
> Auto-generated by prioritize skill. Do not edit manually.
> Last updated: <ISO timestamp>

| Eff. Priority | File | Base | Last Read | Reads | Status |
|---------------|------|------|-----------|-------|--------|
| 100 | SEED.md | 100 | 2026-04-01T18:36Z | 5 | active |
| 95 | layers/prompt.md | 95 | 2026-04-01T18:30Z | 3 | active |
| ...
```

**Status values:** `active` (read within 48h), `stale` (>48h), `new` (never read)

## 2. The `prioritize` Skill

Runs as **step 0** of every `/loop` iteration.

### Scoring Algorithm

```
effective_priority = base_priority × freshness_multiplier

freshness_multiplier:
  read within 1 hour  → 1.0
  read within 24 hours → 0.9
  read within 48 hours → 0.8
  read older than 48h  → 0.7

Floor rule: files with base_priority >= 85 never decay below 70
(core identity/comms files stay hot regardless of freshness)
```

### Procedure

1. Glob `**/*.md` in `.claude/` (recursive)
2. Also check `inbox.md` and `outbox.md` at project root
3. Read each file's frontmatter — extract priority, last_read, read_count
4. Compute effective priority per formula above
5. Sort descending by effective priority
6. Write compiled table to `.claude/priorities.md`

### Skill Output

```
[prioritize] Scored 22 files. Top 5: SEED.md(100), layers/prompt.md(95), journal.md(90), inbox.md(85), layers/intent.md(88→79).
Stale: 3 files (turborepo, evaluation, manifest). New: 0.
```

## 3. Mailbox Relocation

### Before → After

```
.claude/prompt-inbox.md     →  inbox.md   (project root: ./ )
.claude/prompt-outbox.md    →  outbox.md  (project root: ./ )
```

### Rationale

The mailbox is the project's **public interface** — origin writes to inbox, reads from outbox. The `.claude/` folder is **internal governance**. Placing mailbox at project root makes the communication surface immediately visible and consistent with how origin navigates child projects (reads `<project>/inbox.md`, not `<project>/.claude/prompt-inbox.md`).

## 4. Updated Loop Cycle

The loop skill gains a new step 0:

```
Step 0: Prioritize (NEW)
  → Run prioritize skill
  → Regenerate .claude/priorities.md

Step 1: Comms Check (priority 90)
  → Run prompt-fetch on inbox.md (at project root)

Step 2: Local State Check (priority 80)
  → Read priorities.md for scan order
  → Check dispatch-state.json, git status

Step 3: Decide
  → Produce decision paragraph

Step 4: Act
  → Execute recommended action

Step 5: Update State
  → Update comms-cursor, MEMORY.md
  → Send replies via prompt-reply to outbox.md (at project root)
```

## 5. Updated Self-Scan

Self-scan replaces its hardcoded priority table with:

```
1. Read .claude/priorities.md — get current effective priorities
2. Read files in priority order from the registry
3. Produce orientation paragraph
4. If priorities.md is missing or stale → fall back to base priorities in frontmatter
```
